# AGENT_START.md — Главная точка входа агента

> Читать ПЕРВЫМ перед любой работой над проектом AstrumManager.

---

## ОБЯЗАТЕЛЬНЫЙ ПОРЯДОК ЧТЕНИЯ ПАМЯТИ

Перед началом любой работы выполни следующие шаги по порядку:

1. `AGENT_START.md` (этот файл)
2. `replit.md` — стек, архитектура, команды запуска
3. `PROJECT_HISTORY.md` — история этапов и выполненных задач
4. `.agents/memory/MEMORY.md` — индекс всей памяти агента
5. `.agents/memory/project_state.md` — текущий статус, версия, этап
6. `.agents/memory/workflow.md` — текущий рабочий процесс и этапы
7. `.agents/memory/dev_rules.md` — правила разработки (обязательно к исполнению)
8. `.agents/memory/change_log.md` — последние изменения
9. `.agents/memory/audit.md` — найденные и исправленные дефекты

После чтения сообщить пользователю:
- текущий статус проекта
- текущую версию
- последний завершённый этап
- следующий этап

Затем остановиться и ждать задания.

---

## ОБЯЗАТЕЛЬНАЯ ПРОВЕРКА ПЕРЕД ЛЮБЫМ ИЗМЕНЕНИЕМ

До начала работы выполнить и сообщить результат:

1. Текущий HEAD: `git rev-parse --short HEAD`
2. Состояние origin/main: `git ls-remote origin main`
3. Совпадение HEAD и origin/main (ДА / НЕТ)
4. Версия проекта: из `project_state.md`
5. Состояние Git: `git status`
6. Состояние workflow: Telegram Bot / API Server (RUNNING / FAILED)

**До окончания проверки — ничего не изменять.**

---

## ПРАВИЛА РАЗРАБОТКИ (краткий свод)

Полные правила — в `.agents/memory/dev_rules.md`.

- Никогда не изменять код без явного задания от пользователя.
- Каждое изменение сопровождается обновлением файлов памяти.
- После каждого завершённого задания — полный цикл сохранения (см. ниже).
- Не использовать polling на Replit — только webhook.
- Всегда соблюдать структуру модулей из `.agents/memory/bot-structure.md`.
- Инъекция зависимостей — только через `dp["service_name"]` (см. `userservice-injection.md`).
- Никнеймы — только через `build_full_nick(name, role)` (см. `nick-format.md`).

---

## РАЗБИВКА ЗАДАЧИ НА ЭТАПЫ (обязательно)

Любая задача разбивается на этапы перед началом выполнения.

Один этап:
- не более 2–4 файлов,
  ИЛИ
- один логически законченный блок изменений.

После каждого этапа обязательно выполнить полный цикл сохранения.
Только после успешного push переходить к следующему этапу.

---

## УПРАВЛЕНИЕ КРЕДИТАМИ

Если кредиты могут закончиться в середине задачи:
- Не начинать следующий этап.
- Завершить текущий этап.
- Дождаться Replit Checkpoint.
- Выполнить push.
- Отправить отчёт.

Если Replit Checkpoint отсутствует:
- Не выполнять новые изменения.
- Сообщить только:
  «Ожидаю автоматический Replit Checkpoint для завершения цикла сохранения.»

---

## ПОРЯДОК ОБНОВЛЕНИЯ ПАМЯТИ

После каждого завершённого задания автоматически обновить:

- `project_state.md` — после каждого задания
- `change_log.md` — после каждого задания
- `PROJECT_HISTORY.md` — при завершении этапа
- `workflow.md` — при смене этапа
- `audit.md` — при обнаружении или исправлении дефекта
- `replit.md` — при смене версии проекта
- `MEMORY.md` — при добавлении нового topic-файла

Не ждать отдельной команды — обновление памяти является частью каждого задания.

---

## ОБЯЗАТЕЛЬНЫЙ ЦИКЛ СОХРАНЕНИЯ ПОСЛЕ КАЖДОГО ЭТАПА

После ЛЮБОГО изменения проекта Agent ОБЯЗАН выполнить полный цикл:

1. Обновить память проекта (все релевантные файлы в `.agents/memory/`).
2. Выполнить py_compile для всех изменённых .py файлов.
3. Выполнить необходимые тесты (smoke-test: логи webhook — HTTP 200, нет критических ошибок).
4. Проверить запуск всех workflow (Telegram Bot / API Server — статус RUNNING).
5. Убедиться, что Replit Checkpoint создан (платформа создаёт автоматически).
6. Выполнить push в origin/main через `bash scripts/push.sh`.
7. Проверить `git ls-remote origin main` — origin/main == локальный HEAD.
8. Только после п.7 переходить к следующему этапу или считать задание завершённым.
9. Только после п.7 отправлять итоговый отчёт пользователю.

**Если push НЕ выполнен** — запрещено писать «СТАТУС: выполнено».
Необходимо указать причину, что GitHub ещё не обновлён, и что нужно сделать.
