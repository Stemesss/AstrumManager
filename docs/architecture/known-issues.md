# Известные архитектурные проблемы и ограничения

Этот файл фиксирует наблюдаемые проблемы без их исправления.

## 1. Устаревшие и дублирующие точки правды

### 1.1 `bot/webhook.py` устарел

В `bot/webhook.py` прямо указано, что модуль устаревший, а точка входа перенесена в `main.py`.

Риск:

- новый разработчик может принять его за актуальный entrypoint;
- часть runtime-логики уже живёт в другом месте.

### 1.2 `replit.md` частично отстаёт от структуры

Документ содержит полезное описание, но, например, упоминает `bot/config.py`, тогда как актуальный код использует пакет `bot/config/`.

Риск:

- архитектурная документация вне `docs/architecture` может вводить в заблуждение;
- часть сведений приходится перепроверять по коду.

## 2. Смешение активной и заготовочной инфраструктуры

В одном репозитории сосуществуют:

- активный SQLite-слой Python-бота;
- заготовка PostgreSQL/Drizzle в `lib/db`;
- React sandbox;
- OpenAPI-generated пакеты.

Риск:

- без явной документации легко принять за продуктивные компоненты то, что пока является подготовкой или вспомогательным слоем;
- будущие изменения могут случайно затронуть не тот источник истины.

## 3. Высокая связность через dispatcher context

Сервисы и shared values прокидываются через строковые ключи `dp[...]`.

Риск:

- ошибки переименования проявятся только во время выполнения;
- отсутствует единый типобезопасный контракт на набор инжектируемых зависимостей.

## 4. Высокая чувствительность к порядку роутеров

`build_dispatcher()` содержит значимые комментарии о порядке подключения роутеров.

Риск:

- при перестановке роутеров можно сломать cancel-flow, nick onboarding, admin interception и приватные FSM-сценарии без изменения сигнатур функций.

## 5. Концентрация ответственности в `Database`

`bot/database/db.py` объединяет:

- создание схемы;
- миграции;
- CRUD по пользователям;
- CRUD по контенту;
- журнал аудита;
- статистические запросы;
- жалобы;
- вложения;
- темы.

Риск:

- файл является крупной точкой изменения;
- любое будущее перераспределение ответственности потребует аккуратного поэтапного рефакторинга.

## 6. Прямой доступ хендлеров к `Database`

Не все сценарии идут только через сервисы; например, `handlers/complaints.py` и `handlers/publish.py` работают с `Database` напрямую в отдельных местах.

Риск:

- фактический сервисный слой не полностью изолирует хендлеры от хранилища;
- при дальнейшем развитии нужно учитывать уже существующие обходы сервисов.

## 7. Нет явного слоя миграций SQLite

Текущие изменения схемы выполняются через условные `ALTER TABLE` внутри `Database.connect()`.

Риск:

- миграционная история не отделена от runtime;
- при росте схемы поддержка может усложниться.

## 8. Ограничения тестового контура

На момент анализа в репозитории не найдено готовых тестовых файлов, а запуск `python -m pytest` в окружении завершился ошибкой из-за отсутствия `pytest`.

Риск:

- подтверждение поведения сейчас больше опирается на smoke/build checks, чем на автоматизированные тесты;
- перед изменением бизнес-логики в следующих этапах потребуется усиление тестового контура или осторожная поэтапная проверка.
