## Рабочий процесс (SOP)

### Начало задания
1. Прочитать `AGENT_START.md`
2. Прочитать файлы памяти согласно порядку из `AGENT_START.md`
3. Выполнить обязательную проверку состояния (см. ниже)
4. Сообщить пользователю текущий статус
5. Ждать задания

### ОБЯЗАТЕЛЬНАЯ ПРОВЕРКА ПЕРЕД ЛЮБЫМ ИЗМЕНЕНИЕМ

До начала работы выполнить и сообщить:
```
1. git rev-parse --short HEAD          — текущий HEAD
2. git ls-remote origin main           — состояние origin/main
3. Совпадение HEAD и origin/main       — ДА / НЕТ
4. Версия проекта из project_state.md
5. git status                          — состояние рабочего дерева
6. Статус workflow: Telegram Bot / API Server (RUNNING / FAILED)
```
До окончания проверки — ничего не изменять.

### Выполнение задания
1. Проанализировать задание
2. Уточнить неясности (если есть)
3. Разбить задание на этапы (не более 2–4 файлов на этап или один логический блок)
4. Выполнять этапы последовательно — после каждого: полный цикл сохранения
5. Только после успешного push переходить к следующему этапу
6. Вывести итоговый отчёт в стандартном формате после финального этапа

### УПРАВЛЕНИЕ КРЕДИТАМИ
Если кредиты могут закончиться в середине задачи:
- Не начинать следующий этап
- Завершить текущий этап
- Дождаться Replit Checkpoint
- Выполнить push
- Отправить отчёт

Если Replit Checkpoint отсутствует:
- Не выполнять новые изменения
- Сообщить: «Ожидаю автоматический Replit Checkpoint для завершения цикла сохранения.»

### ОБЯЗАТЕЛЬНЫЙ ЦИКЛ СОХРАНЕНИЯ (после каждого этапа)

```
1. Обновить память проекта (project_state.md, change_log.md, etc.)
2. py_compile для всех изменённых .py файлов
3. Smoke-test: проверить логи webhook — HTTP 200, нет критических ошибок
4. Проверить все workflow (Telegram Bot / API Server — статус RUNNING)
5. Убедиться в создании Replit Checkpoint (платформа создаёт автоматически)
6. bash scripts/push.sh — push в origin/main
7. git ls-remote origin main — убедиться: origin/main == локальный HEAD
8. Только после п.7 — переходить к следующему этапу или считать задание завершённым
9. Только после п.7 — отправлять итоговый отчёт
```

**Если push НЕ выполнен** — запрещено писать «СТАТУС: выполнено».

### После каждого задания обязательно обновить
- `project_state.md` — статус, последнее задание
- `change_log.md` — запись об изменении
- `audit.md` — если найден/исправлен дефект
- `workflow.md` — если изменился этап
- `PROJECT_HISTORY.md` — если завершён крупный этап
- `replit.md` — если изменилась версия

## Git-процесс

- git commit/add заблокированы в main agent — коммит = Replit Checkpoint (автоматически).
- Push выполняется через `bash scripts/push.sh` (поддерживает GITHUB_TOKEN и AstrumManagerMain).
- ВАЖНО: в main agent заблокированы `git add`, `git commit`, `git reset`.
  Push возможен только без force-флагов.
  (см. `.agents/memory/git-ops.md`)
