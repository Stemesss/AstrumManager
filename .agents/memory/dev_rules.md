# Правила разработки AstrumManager

## 1. Общие правила

- Не изменять код без явного задания от пользователя.
- Не создавать новые файлы без необходимости.
- Всегда сохранять существующую структуру модулей.
- Не использовать глобальные переменные — только DI через `dp["name"]`.
- Комментарии в коде — на русском языке (в соответствии с проектом).

## 2. Архитектурные правила

### Webhook
- На Replit и Railway — только webhook-режим.
- Polling запрещён (Replit блокирует исходящие на api.telegram.org).
- Авто-детекция: `WEBHOOK_BASE_URL` → `REPLIT_DOMAINS` → `RAILWAY_PUBLIC_DOMAIN` → polling (только local).

### Два процесса
- Python-бот (aiohttp): порт 6000, только для внутренних запросов.
- Node.js Express: порт 8080, публичный прокси.
- Express ДОЛЖЕН пропускать `/api/telegram` без `json()` парсинга тела.

### База данных
- Только SQLite через aiosqlite.
- WAL-режим обязателен (`PRAGMA journal_mode=WAL`).
- Все операции — через `Database`-класс (`bot/database/db.py`).
- Перед сбросом сезона — создать бэкап (`db.create_backup()`).

## 3. Правила по коду

### Инъекция зависимостей
```python
# В build_dispatcher() в main.py:
dp["user_service"] = UserService(db)

# В handler:
async def handler(message: Message, user_service: UserService) -> None:
    ...
```
Список инъектируемых объектов: `user_service`, `audit_service`, `stats_service`,
`news_service`, `topic_service`, `db`, `group_chat_id`, `owner_id`.

### Ники пользователей
- `game_nick` в БД хранит ТОЛЬКО имя (например: "Вадим").
- Полный ник = `build_full_nick(name, role)` → "✪ Вадим" (символ + имя, без слова-титула, без разделителя).
- Нельзя хранить полный ник в БД — он вычисляется на лету.

### Роли (единая схема символов, v1.2.8+)
```
LEADER     → "✪ Лидер"
CLAN_CHILD → "✦ Дитя клана"
ELDER      → "✧ Старейшина"
MEMBER     → "◇ Участник"
```
Источник истины — `bot.utils.roles.ROLE_DISPLAY_ICONS`. Никаких "Рекрут"/"Воин" — эти
легаси-названия полностью удалены из проекта (регистрация, профиль, участники, превью,
сообщения, custom_title, шаблоны, уведомления).

### Защита пользователей
- Нельзя удалить суперпользователя (owner_id = 8490615925).
- Нельзя удалить себя.
- Нельзя удалить пользователя с ролью LEADER.

### Очки активности
Вычисляются on-the-fly из `audit_log` (не хранятся в отдельной таблице):
```
news_create       = 5
guide_create      = 10
screenshot_upload = 2
event_create      = 8
```

## 4. Порядок регистрации роутеров

```python
# В main.py, private router:
cancel → common → setrole → nick → audit → news → publish → topics →
icons → members → content → complaints → rules → admin → stats →
statistics → menu → echo
```

Порядок важен — более специфичные обработчики должны идти раньше.

## 5. Правила памяти агента

- После каждого задания — обновить файлы памяти (не ждать команды).
- Не сохранять в память: секреты, пути к файлам, имена функций, номера строк.
- Сохранять в память: архитектурные решения, нетривиальные ограничения, паттерны.
- MEMORY.md — только индекс (одна строка на файл).
- topic-файлы — детальное описание.

## 6. Правила Git и обязательный цикл сохранения

- git commit/add заблокированы в main agent — коммит создаётся автоматически (checkpoint).
- git push без force-флагов — разрешён через `bash scripts/push.sh`.
- Сообщение коммита — осмысленное, на английском.
- Все файлы памяти (.agents/memory/) должны попадать в Git.
- AGENT_START.md и PROJECT_HISTORY.md — тоже в Git.

### Обязательный цикл после каждого задания:
1. Обновить память проекта.
2. Smoke-test (проверить логи — HTTP 200, нет критических ошибок).
3. Проверить все три workflow (RUNNING).
4. Дождаться/убедиться в создании Replit Checkpoint.
5. `bash scripts/push.sh` — push в origin/main.
6. `git ls-remote origin main` — убедиться, что GitHub обновлён.
7. Только после п.6 считать задачу завершённой и отправлять отчёт.

**Если push НЕ выполнен** — запрещено писать «СТАТУС: выполнено».

### push.sh — поддерживаемые токены:
- `GITHUB_TOKEN` (приоритет)
- `AstrumManagerMain` (fallback)
Оба имени поддерживаются автоматически. Ручная настройка не требуется.

## 7. Правила тестирования

- После каждого изменения — проверить логи воркфлоу.
- Ошибка "TelegramBadRequest: chat not found" после smoke-тестов — ожидаема, не дефект.
- Реальное тестирование — отправкой сообщений боту.

## 8. Формат итогового отчёта (обязательный, постоянный)

- Итоговый отчёт — ВСЕГДА один блок ```text``` (для кнопки «Копировать» в интерфейсе Replit).
- Строго фиксированные поля, в этом порядке, без изменений названий:
```
ЗАДАНИЕ:
СТАТУС:
ВЕРСИЯ:
ЭТАП:
ИЗМЕНЕНИЯ:
ФАЙЛЫ:
ПРОВЕРКА:
ТЕСТЫ:
WORKFLOW:
GIT:
GITHUB:
ПАМЯТЬ:
ИТОГ:
```
- Запрещено: markdown-таблицы, HTML-таблицы, списки сравнения (X | Y) и другие длинные структурированные конструкции внутри отчёта.
- Правило действует для ЛЮБОГО задания, включая диагностику, проверку, read-only анализ — без исключений.
- Внутри полей допускается обычный текст и короткие перечисления через тире/запятую, но не таблицы.

## 10. Формат ответов агента (все ответы, без исключений)

- Все ответы — только обычный текст.
- Все отчёты, планы, задания, списки — исключительно в одном блоке ```text``` с кнопкой «Скопировать».
- Запрещены: карточки Replit, Plan, Canvas, интерактивные блоки, таблицы, раскрывающиеся списки, любые форматы, которые нельзя скопировать целиком одним нажатием.
- Правило действует для ЛЮБОГО ответа — задания, диагностика, планы, пояснения, read-only анализ.

## 11. Регламент разработки (постоянный, обязательный)

### Проверка перед началом работы
Перед любым изменением выполнить и сообщить результат:
- git rev-parse --short HEAD
- git ls-remote origin main
- Совпадение HEAD и origin/main (ДА / НЕТ)
- Версия проекта из project_state.md
- git status
- Статус workflow: Telegram Bot / API Server
До окончания проверки — ничего не изменять.

### Разбивка задачи на подэтапы и логические блоки
Любая задача разбивается на подэтапы перед выполнением (для планирования).
Подэтапы выполняются строго последовательно.
Цикл сохранения выполняется не после каждого подэтапа, а после завершения
одного логического блока.

Логический блок:
- 2–3 связанных подэтапа, ИЛИ
- не более 6 изменённых файлов за блок (суммарно по подэтапам блока).

### Цикл сохранения после каждого логического блока
1. py_compile для всех изменённых .py файлов
2. Необходимые тесты (smoke-test)
3. Проверка workflow (RUNNING)
4. Ожидание Replit Checkpoint
5. bash scripts/push.sh
6. git ls-remote origin main — origin/main == локальный HEAD
Только после п.6 переходить к следующему блоку.
Внутри блока промежуточный push/checkpoint не выполняется.

### Управление кредитами
Если кредиты могут закончиться:
- не начинать следующий подэтап/блок
- завершить текущий подэтап, выполнить полный цикл сохранения (даже если блок
  не закрыт), дождаться Checkpoint, выполнить push, отправить отчёт

Если Checkpoint отсутствует:
- не выполнять новые изменения
- сообщить: «Ожидаю автоматический Replit Checkpoint для завершения цикла сохранения.»

## 12. Регламент взаимодействия с пользователем (постоянный, обязательный)

### Правило 1 — Обязательный план перед крупными задачами
Перед любой задачей, которая изменяет более 4 файлов, состоит из нескольких
логических частей или требует изменения архитектуры — агент ОБЯЗАН сначала
показать план (название, цель, файлы, результат, риски для каждого этапа).
После вывода плана — остановиться и ждать подтверждения пользователя.

### Правило 2 — Приоритет стабильности
Если обнаружена ошибка, регрессия или нестабильная работа — запрещается
продолжать новый функционал. Сначала: причина → исправление → проверка.

### Правило 3 — Варианты реализации на согласование
При наличии нескольких вариантов реализации — предложить пользователю варианты.
Не принимать архитектурные решения самостоятельно.

### Правило 4 — Только функциональные предложения
Предложения по развитию — только функции AstrumManager.
Не предлагать Deploy, Git, Environment, Infrastructure без явного запроса.

### Правило 5 — Запрет дублирования
Перед новой функцией — проверить наличие похожего функционала.
Если существует — расширять, не дублировать.
Запрещены: второй обработчик, второе меню, второй сервис с одинаковой логикой.

### Правило 6 — Формат ответов
Все ответы — обычный текст, один блок text с кнопкой «Скопировать».
Запрещены: карточки, Canvas, Plan, таблицы, интерактивные блоки.

### Правило 7 — Обязательный отчёт
После каждого этапа — полный отчёт в установленном формате без пропуска разделов.

### Правило 8 — Запрет «СТАТУС: выполнено» без GitHub
Запрещено до тех пор, пока origin/main не совпадает с локальным HEAD.

## 9. Запрещено

- Использовать polling на Replit/Railway.
- Хранить полный ник в БД.
- Использовать слова "Рекрут"/"Воин" или иные легаси-титулы ролей — только ✪/✦/✧/◇.
- Изменять `owner_id` (8490615925) в коде.
- Добавлять `json()` middleware для `/api/telegram` путей в Express.
- Создавать глобальные синглтоны сервисов.
- Писать «СТАТУС: выполнено» если push в GitHub не был выполнен.
